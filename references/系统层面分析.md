## 一、一句话诊断

现有架构是一台**"真人古装短剧成片机"**，设计隐含假设停留在 **2023–2024 的付费小程序时代**；而 2026 行业现实已是 **免费(IAA)主导、AI 漫剧领跑、变现靠平台分账/完播率、监管系统性收紧**。系统造得出"片子"，但缺了把片子变成钱的整条**变现回路**与**合规闸门**。

## 二、错位诊断（现有架构的隐含假设 vs 2026 现实）

| # | 现有架构隐含假设 | 2026 行业现实 | 错位 |
|---|---|---|---|
| 1 | 优化目标 = **成片好不好看**（visual_qa 查画面、director 终审）[arch §二] | 免费模型看**完播率/追剧率**，付费看**素材 CTR/当日 ROI** [行业§4.1/§2.1] | **没有任何变现指标回路**——系统不知道片子卖不卖得动 |
| 2 | 形态 = **真人**（即梦文生图 + 可灵图生视频）[arch §九] | AI 变现先锋是**漫剧**（动画/仿真人），成本 1/10、开辟男频增量 [行业§十三] | 押在最内卷、成本最高的真人赛道 |
| 3 | 链路终点 = **07_成片/ep.mp4**（compose 完即结束）[arch §三] | 成片只是中间品，下游还有**投流素材裂变 + 跨平台分发(抖音→红果)** [行业§六/§8.2] | 缺整条变现链，做完即"断头" |
| 4 | 质检 = 画面质量（穿模/手指）[arch §六] | 监管要**AI 标识三处 + 六大红线 + 备案**，否则限流/下架（过审率仅 30%）[行业§十一] | **无合规闸门**，按现状产出大概率过不了审 |
| 5 | 角色一致性 = 被动传 `reference_images`[arch §十三] | 角色一致性是 2026 工业化**命门**，配合**抽卡择优** [行业§6.1] | 一致性靠运气，无主动保证/择优 |
| 6 | 串行执行（asyncio 未做）[CLAUDE] | 工业化靠**批量产能**（巨日禄 3人15天80集、九州单月1399部）[行业§十二] | 产能上不去，规模化无从谈起 |

> **结论**：错位 1、3、4 是**结构性缺口**（缺整块能力），错位 2、5、6 是**方向/工程问题**（可调可补）。

## 三、战略路线选择（决定后续一切）

三条路，系统改造方向完全不同：

| 路线 | 内容形态 | 变现 | 优化目标 | 系统改造重心 | trade-off |
|---|---|---|---|---|---|
| **A. 国内免费 + AI 漫剧**（推荐） | AI 漫剧（动画/仿真人） | 红果/番茄分账 | 完播率/追剧率 | 漫剧生成引擎 + 角色一致性 + 完播钩子 + 合规标识 | ARPU 低(8-12元)、单价暴跌(§11.4)、监管严；红海化的蓝海 |
| B. 出海付费 + 仿真人 | AI 仿真人短剧 | ReelShort 阵营 IAP | 素材 CTR/当日 ROI | 投流素材裂变 + 本土化题材 + 多语言 | 北美 RPD 4.7 美元、ARPU 高，但 CPI 高、九成亏、头部双雄占四成 |
| C. 国内付费小程序（现状默认） | 真人/仿真人 | 自投千川赌 ROI | 当日 ROI | 投流素材 + ROI 回路 | 付费侧低迷、转化率跌到 3-5%、窗口收窄 [行业§2.1] |

### 已选：A+B（先 A 跑通，再复用扩 B）

**理由（全部可回溯到行业报告）：**
1. **AI 成本优势咬合免费走量**：免费侧靠量，AI 批量产能正适合喂平台内容饥渴 [行业§六小结]。
2. **男频漫剧是增量蓝海**：免费盘女频占 70%，漫剧男性占 90%，补的是缺口而非抢存量 [行业§13.2]。
3. **与现有系统能力最近**：现有 t2i/i2v 链路改造成漫剧引擎，比从头做出海多语言/IAP 体系工程量小。
4. **B 作为第二阶段**：海外付费 ARPU 高（北美 RPD 4.7 美元），用 A 沉淀的底层能力扩张，单位经济更好 [行业§9.2]。

**诚实 trade-off**：A 的天花板被 ARPU 低 + 单价暴跌(30-100→5-10元) + 监管整治压着，是"红海化的蓝海"；B 门槛在多语言/本土化/海外投放。两阶段法的价值在于：**先用 A 验证工业化与合规、把固定成本摊薄，再用极低边际成本扩 B**（见 §3bis）。

## 三bis、A→B 桥接分析（A+B 路线的关键）

A+B 不是"同一批内容投两个市场"那么简单。调研后有三层结论：

**① 好消息：译制层已极度便宜，"一片多语言"工作流成熟**
- AI 译制让单部本地化成本从 ~4000 元降至 **~300 元**（降 90%+），周期 3 天→1 天（Sereal+，知乎 2026）；AI+人工审校混合模式成本降 **72%+**、效率提升 **85%**（腾讯云）。
- 阿里云视频云等可做到"**制作单一视频 → AI 自动输出多语言多角色**"；工具链成熟（ElevenLabs / RaskAI / 趣丸千音 / Vozo / 鬼手剪辑）。
- → **A 的成片复用到 B 的边际成本极低**，这是 A+B 成立的工程基础。

**② 坏消息（关键诚实修正）：AI 漫剧出海当前收益弱于真人**
- 36氪/投中网：AI 漫剧/AI 短剧出海"**难言乐观**"，在 YouTube/垂直短剧 App 的收益**远低于真人短剧**，海外仍处探索阶段。
- 对比：漫剧**国内**2026 预计 **350–500 亿**、增速领跑真人；但**出海定量收益数据缺失**（只有定性"远低于真人"，一手数字未公开——已记入行业文档待核实）。
- 海外大盘锚点：2025 海外收入约 **20.58 亿美元**（点点数据）/ **23.8 亿美元**（光大证券，口径偏高）；昆仑万维短剧平台 2025-12 月流水近 **3600 万美元**、ARR 超 4 亿美元（可见高 ARPU 主要由真人/仿真人贡献）。
- → 即"国内男频漫剧直接译制出海"未必能跑动 B 的高 ARPU；**B 的最优形态可能是仿真人/真人译制，而非漫剧**。

**③ 机会与差异：漫剧有跨文化优势，海外要差异化题材**
- 漫剧对海外用户"**比国内真人短剧更好理解**"（动画无演员/文化违和），有增长潜力但收益未被验证（知乎《20万成本北美2亿》）。
- 东南亚等差异化题材看好**恐怖/黑帮/BL/校园爱情/轻喜**（中国稀缺品类）[行业§九]。

**桥接结论（修正 A+B 的含义）：**
> A→B 真正可复用的是**底层工业化能力**（生成引擎、角色一致性、抽卡择优、合规工作流）+ **译制层**，**而非内容本身一定通用**。务实做法：
> - **A 阶段**：国内免费男频漫剧，把工业化+合规跑通；
> - **B 阶段**：复用引擎与译制层，但**内容形态/题材按海外重做或转仿真人**（漫剧出海作为低成本试验田，验证后再加码，不押全部）。
> 这样 §五 的模块里，**译制/多语言层**是 A→B 的桥（边际成本低、优先级可后置到 B 启动前），而生成引擎/角色一致性/合规是 A、B 共用的地基（P0）。

## 五、建议的改造/新增模块（按优先级）

**P0（不做则产出过不了审 / 质量没下限）**
1. **合规闸门**（新增）：compose 后强制加 AI 标识三处；project.yaml 加备案号字段；visual_qa 扩一个"红线审查"子任务（六大红线，尤其商三官是复仇题材→盯"极端复仇"尺度）。
2. **漫剧生成引擎 + 角色一致性**：text2img 换漫剧风格 provider；角色卡升级为"一致性锚"（固定形象 embedding/参考图集），t2i 强制带锚。
3. **抽卡择优**：text2img 一个 shot 生成 N 版 → visual_qa 自动选最佳（复用现有 qa，加"多版打分择优"）。

**P1（决定质量上限与产能）**
4. writer/storyboard prompt 按**完播率/追剧钩子**重写（钩子前置、集间续接、男频强设定爽点）。
5. 接真实 img2video provider + 音画同步。
6. 落地 asyncio 并行（arch §五已设计、§十四之并行约束已写明，待实现）。

**P2（A 变现闭环，跑通后再上）**
7. **分账/变现换算**：state 加 `monetization` 字段，把"0.2元/分钟×有效时长"[行业§5.2] 做成收益估算。
8. **完播率反馈回路**：新增 executor 拉平台数据 → 写回 state → 反哺创意层（错位 #1 的解药）。

**P3（B 出海阶段，A 跑通后启动；多数复用 A 的地基）**
9. **译制/多语言层**（新增 executor）：单一成片 → AI 多语言字幕+配音+口型对齐（成本~300元/部、工具成熟，见 §3bis）。A→B 的桥。
10. **投流素材（drama 内嵌模块，源头式）**：在 storyboard 阶段标注钩子/卡点元数据，成片时导出"带元数据的素材原料"。**注意边界**：这是**源头式、语义驱动**的内嵌模块，只服务自家剧、不碰投放；与之相对的"**面向投手的爆款情报+复刻平台**"（事后式、数据/代理驱动、服务多内容源）是**另一个独立系统**，分析见 `references/投手素材平台分析.md`，两者仅以数据契约对接、保持系统独立。
11. **当日 ROI 回路 + 海外题材/形态适配**：B 优化目标是 CTR/当日 ROI；内容按 §3bis 结论可能需转仿真人或换差异化题材，非漫剧直接译制。

## 七、交互设计（人机交互方案）【2026-06-26 新增】

> 回答"这个系统该怎么和人交互"。当前现状：CLI 批处理 + 文件式异步人审（`FileReviewChannel`，靠重跑 `--review-reply` 续）+ 可选 `Notifier` 通知；无 GUI、无实时对话、Hermes 入口默认关且坏。

### 7.1 核心原则：异常驱动（exception-based）

系统是**跑几小时到几天的长时批处理**，人不可能守终端；且需人介入的点**稀疏且要看画面**。所以交互设计的第一原则是**默认全自动，人只在异常/关卡被"推"到**：
1. 配置的人审关卡（选题 / 剧本 / 终审，`stage_modes: review`）；
2. 升级镜头（重试耗尽 `escalated`）；
3. 成本/错误预警。
其余交给 visual_qa / director 自动判——**绝不让人逐镜头点**（80 集 ×~18 镜头，逐个确认 = 自动化白做）。这与现有设计意图一致。

### 7.2 两层交互

**① 运维层 = CLI（保留）**：`init / run / status / --stage / --verbose`，用于启动、调试、批处理。开发期主力，不砍。

**② 人在环层 = IM 机器人（建议新增，日常交互主场）**：推荐 Telegram 起步（或飞书/企微）：
- **推送**：待审包（带**缩略图/片段**）、升级镜头、成本预警、完成通知；
- **回复闭环**：IM 内直接回"通过 / 打回 镜头3 …"，机器人解析 → 写状态 → **自动续跑**（免回终端敲 `--review-reply`）；
- **查询**：`status ep03` 拿进度。

**为什么 IM 而非网页**：决策点稀疏、时移、在手机上、**审核要看图**——CLI 都满足不了，IM 天然满足。这正是当初研究 Hermes 想要、但其不开箱支持的"**主动找人 → 多轮聊 → 结论回传**"。

### 7.3 架构已预备好（缺的只是一个实现）

- `ReviewChannel` 抽象（post/poll/ack）已在，`FileReviewChannel` 为文件实现，文档早已注明"Telegram/IM 作为后续可插拔 provider"——**新增一个 `TelegramReviewChannel`（既推送又收回复）即可**，不动调度核心。
- `Notifier` 已是多渠道结构，IM 通知顺势接入。

### 7.4 必须配套的前置改动：运行形态

现 orchestrator 是**一次性"挂起即退出"**，IM 异步回复续不上。要让"回复 → 自动续跑"成立，需二选一：
- (a) orchestrator **常驻守护进程**：轮询 review channel，有回复就推进（更干净，长期推荐）；
- (b) 轻量 bot 服务：收到回复时触发 `--review-reply` + 续跑（改动小，过渡方案）。
> 这是把交互做顺的关键前置，属**非平凡改动**。

### 7.5 可选：只读看板（后置）

进度/成本/镜头画廊/待审队列的 web dashboard——**单人运营 IM 已够**，团队/多项目再上，勿过早做。

### 7.6 交互相关的优先级与收尾

1. `TelegramReviewChannel`（填人审闭环，复用现有抽象）
2. orchestrator 守护进程化（让续跑自动）
3. （可选）只读 dashboard
4. **清理坏的 Hermes 入口**：`HermesNotifier` 默认关、端点(`/api/notify`)与端口(7777)均错，既走 IM 应删（见 `notify.py` / `config.py` / `config.yaml`）。

> honesty：本节为**推荐方案**，非代码现状；守护进程化非小工程。当前阶段仅设计，不改代码。
