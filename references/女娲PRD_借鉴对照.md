# 女娲 PRD v2.2 ↔ 当前系统 借鉴对照

> 整理日期 2026-06-21。对照对象:`references/女娲 PRD v2.2 (离线版).html`(用户在 Claude Design 设计的多 Agent 短剧平台 PRD)vs 当前 `drama/` 制作引擎。
> 目的:挑出能借鉴/融合到当前引擎的设计,并诚实划清哪些不该塞。

## 0. 定性:两者不在一个层

- **女娲 = 平台 / SaaS 产品愿景**:多租户、公司后台、双角色权限、算力计费、工作流模板、分发治理、平台运营后台。重心在"一家短剧公司怎么运营"。
- **当前系统 = 单机制作引擎**:orchestrator + agents + executors,信条"系统是核心、项目是数据"。重心在"单部剧怎么做出来、质量怎么保证"。

**正确关系**:当前引擎可作为女娲的"执行内核"。借鉴只取**落在引擎层**的设计;平台层(权限/计费/分发/审计)不塞进引擎,那是未来若上平台时套在外层的东西。

> 互补提醒:女娲 PRD 篇幅一多半在运营治理(权限/计费/分发),**单集制作质量保证**反而较薄——而那正是本引擎的核心。融合时别让平台层盖过"先把一集做好"。

## 1. 🟢 强烈建议借鉴(引擎层 + 补真短板)

### 1.1 六阶段:美术独立成阶段,与剧本并行
- 女娲:从「5 阶段」扩成「6 阶段」,把**美术**从剧本剖分为独立阶段与独立角色(丹青/校境),**与剧本并行**,二者同为分镜的前置条件。美术产出 立绘 + LoRA + 道具 + 场景 + 美术包,自动入资产库。
- 我方现状:只有 script→storyboard,**无独立美术阶段**;角色一致性所需的 LoRA/参考图无正式产出归属。
- 价值:这是**角色一致性的归宿**(呼应 `references/模型选型_2026-06.md` 的 LoRA 路线)。
- 落地形态:新增 stage `art`(在 script 之后、storyboard 之前;storyboard 解锁条件 = script done AND art done)。art 产出角色立绘/LoRA 引用,写入 state,供 text2img/img2video 继承。

### 1.2 三种协作模式(每环节独立选)
- 女娲:每个角色/环节独立选 **真人主控 / 仅审核 / 全自动**;支持"一键自动生成"= 跑所有 ②③ 模式环节,①模式留给真人手动。
- 我方现状:写死"全自动",且离线下 QA/导演是橡皮图章(永远通过)。
- 价值:直接回答"是不是全自动"——应让**每个 stage 可配 mode**。
- 落地形态:`project.yaml` 每 stage 加 `mode: auto | review | manual`;orchestrator 据此决定该环节是否自动跑、是否需真人审核、是否暂停等人工。

### 1.3 执行 Agent + 审核 Agent 分离 + 抽卡选优
- 女娲:执行 Agent 担任角色槽位;审核 Agent 把关——**抽卡步骤选优、非抽卡步骤判过审**。
- 我方现状:有 visual_qa/director 审核,但流程是"生成 1 个 → 过/不过 → 重试"。
- 价值:"生成 N 候选 → 审核选最优"(抽卡)比"生成 1 个赌过不过"成片率高得多。
- 落地形态:text2img/img2video 支持产出 N 个候选;visual_qa 从"判过/不过"升级为"在 N 个里选最优 + 给分"。

## 2. 🟡 可轻量借鉴(小改、性价比高)

### 2.1 成本拆账"创作 vs 审核"两段
- 女娲核心创新:把每笔消耗拆成 创作(执行 Agent)+ 审核(审核 Agent)两段;五个拆账视图(按阶段 / 创X审Y / 剧预算 / 成员 / 单价)。
- 我方现状:已按环节记账(见 VIBE_CODING_LOG 2026-06-21 成本记账条),但未分创作/审核桶。
- 落地形态:cost_tracker 给每个环节标 `bucket: create|review`(writer/storyboard/art→create,visual_qa/director→review),summary 增"创作 vs 审核"占比。**这是接上已完成记账的最小增量。**

### 2.2 工作流模板
- 女娲:六阶段(协作模式 + 执行/审核 Agent + 技法装备 + 抽卡次数)整体存为模板,新建作品一键套用,模板携带每集预估算力/工时。
- 落地形态:`project.yaml` 的 `production`(含 1.2 的 per-stage mode、重试预算、抽卡次数)抽成可复用模板文件,新项目引用。

### 2.3 LoRA 资产持久化 + 跨环节继承
- 女娲:LoRA 训练产出必须持久化为资产(版本号/训练集hash/评分/源剧ID/耗点);分镜挂载的 LoRA **自动继承到视频环节**;装备权重按景别智能推荐。
- 落地形态:shot 状态加 `lora`(引用 + 权重);img2video 自动继承 storyboard/art 阶段定的 LoRA。

## 3. 🔴 不建议塞进引擎(诚实划界)

以下是**多租户平台功能**,与单机引擎不在一层,塞入会污染"系统是核心、项目是数据":
- 公司后台 / 客户自管理空间
- 双角色权限模型(frontRoles + backAdmin)
- SaaS 算力计费 / 充值 / 预算告警 / 套餐(注:**计费的"按环节/创作审核拆账"口径**可借鉴到记账,见 2.1;但充值/订阅/多租户账户体系不进引擎)
- 内容分发渠道(抖音/视频号/快手上架)
- 审计流水 / 平台运营后台
正确位置:未来若把引擎包装成平台,这些套在引擎**外层**。

## 5. 建议落地优先级

1. **协作模式分级**(1.2)—— 回答"全自动程度",抽象干净,改动集中在 orchestrator + project.yaml
2. **美术独立阶段**(1.1)—— 角色一致性的归宿,与 LoRA 选型呼应
3. **成本拆账创作/审核**(2.1)—— 接已完成记账的最小增量
